## 我的选择：Original Model + ALNS

### 为什么选 Original Model

Reformulation 模型引入 λ_ivt 变量，核心价值在于：将收集点库存约束转化为最短路网络结构，从而使 LP 松弛更紧，服务于 LBBD 中的列生成和对偶割生成。但在启发式算法中，我们直接操作可行解而非求解 LP 松弛，这些优势消失了。

Original Model 的变量 z_it（是否访问）、q_it（取货量）、x_ijt（路径弧）、p_t（生产量）具有直接的物理含义，邻域操作（插入/移除一次访问、交换路径、调整生产量）的定义和可行性检查都更直观。而 λ_ivt 的最短路流平衡约束（源节点→汇节点的链式结构）在每次邻域移动后都需要维护一致性，增加了实现复杂度却没有带来解质量的提升。

### 为什么选 ALNS

这个问题是 Multi-Vehicle Production Routing Problem，耦合了三个子结构：lot-sizing（生产批量决策）、inventory management（多层库存管理）、CVRP（多车路径规划）。ALNS 的 destroy-repair 框架天然适合这种多维决策问题，原因有三：

第一，可以针对不同子结构设计专门的 destroy/repair 算子。比如"移除某个收集点在某个时期的访问"是库存层面的 destroy，"重新优化某个时期的路径"是路径层面的 repair，"重新规划一段时间窗内的生产计划"是生产层面的 destroy。这些算子可以独立设计、组合使用。

第二，ALNS 的自适应权重机制会自动学习哪些算子组合在当前问题实例上最有效，比 Tabu Search 的固定邻域结构更灵活。

第三，IRP/PRP 文献中，ALNS 已经是最主流的方法之一，有大量成熟的算子设计可以借鉴。

## 实现思路

### 整体架构

算法分两层：**外层 ALNS 框架**控制迭代、选择算子、接受准则；**内层**包含一组 destroy 算子、repair 算子、以及一个快速可行性检查/评估模块。

### 解的表示

一个完整解 S 包含以下组件：

**访问计划** `z[i][t]`：每个收集点 i 在每个时期 t 是否被访问。**取货量** `q[i][t]`：访问时全量取走（由 z 决定，可以直接计算）。**路径方案** `routes[t]`：每个时期 t 的一组车辆路径，每条路径是一个收集点序列。**生产计划** `y[t], p[t]`：每个时期是否生产及生产量。**库存状态** `I_i[t], I_0[t], P_0[t]`：各类库存，由上述决策完全确定。

关键点：z 和 routes 确定后，q 可以直接算出（访问时清空库存）；给定 q 后，工厂原料库存 I_0 由库存平衡确定；生产计划 p_t 和成品库存 P_0t 则需要单独优化或贪心确定。

### 初始解构造

采用**周期性访问 + 贪心插入**策略：

1. 对每个收集点 i，根据其供给率 s_it 和库存容量 L_i，计算一个最大访问间隔 Δ_i = ⌊L_i / avg(s_it)⌋，然后均匀分配访问时期。
2. 对每个时期 t，将该期需要访问的收集点集合构造 CVRP 初始解——用 Clarke-Wright savings 算法或最近邻插入启发式。
3. 给定所有 q_it 后，用一个简单的 lot-sizing 贪心（或直接解一个小规模 LP）确定生产计划 p_t 和 y_t，使成品库存平衡和工厂原料库存约束满足。

### Destroy 算子（6个）

**D1 - Random Visit Removal**：随机选择若干个 (i,t) 对，将 z[i][t] 从 1 改为 0（移除访问），从路径中删除该收集点。被移除的访问意味着库存会在该期继续累积，需要在之后某期补回。

**D2 - Worst Cost Removal**：计算每个访问 (i,t) 的"边际成本"（该访问带来的路径成本增量 + 该期库存减少带来的库存成本节省），移除边际成本最高的若干个访问。

**D3 - Related Visit Removal**：选一个种子收集点 i₀，移除与 i₀ 地理距离近的收集点在同一时期或相邻时期的访问。这创造了一个局部"空白"，便于 repair 时重新优化。

**D4 - Period Destruction**：随机选择 1~2 个时期，完全清除这些时期的所有路径和访问计划（但保留其他时期不变）。

**D5 - Route Removal**：在某个时期 t 中，随机移除一条完整路径上的所有访问。

**D6 - Production Perturbation**：不改变路径，而是打乱生产计划——随机将某些时期的 y_t 从 1 改为 0（取消生产），迫使生产重新分配。

### Repair 算子（4个）

**R1 - Greedy Visit Insertion**：对所有"未被服务"的需求（因 destroy 导致库存会溢出的收集点），按照溢出紧迫度排序，贪心地在最便宜的时期和路径位置插入访问。插入时检查车辆容量约束。

**R2 - Regret-k Insertion**：计算每个待插入访问的"regret值"（最好插入位置和第 k 好插入位置的成本差），优先插入 regret 值最大的访问，避免贪心陷阱。

**R3 - Period-by-Period Reconstruction**：对被 destroy 的时期，按时间顺序逐期重建——确定该期需要访问哪些收集点（基于库存溢出紧迫度），然后用 savings 算法构造路径。

**R4 - Production Re-optimization**：给定 repair 后的访问计划和路径，将生产计划视为一个确定性 lot-sizing 问题（线性规划），精确求解 p_t, y_t, P_0t, I_0t。这一步计算量很小（变量数 = O(|T|)），可以用单纯形法瞬间解出。

### 评估与可行性维护

每次 destroy-repair 后，需要检查：收集点库存不超容量 L_i（硬约束）、工厂原料和成品库存不超容量（硬约束）、每期车辆数不超过 K、每条路径载重不超过 Q。

关键加速：维护一个**增量评估**机制。对库存，维护前缀和使得修改一个 z[i][t] 后能 O(|T|) 更新该收集点的整条库存轨迹。对路径成本，维护邻接矩阵使得单次插入/删除的成本变化可以 O(1) 算出。

### 接受准则与自适应权重

使用**模拟退火**接受准则：以概率 exp(-ΔZ / temperature) 接受恶化解，temperature 按指数衰减。

自适应权重更新：每个 destroy 算子和 repair 算子维护一个权重 w_d, w_r，每隔一个 segment（比如 100 次迭代）根据表现更新。表现指标：找到新全局最优得 σ₁ 分，找到比当前解好的解得 σ₂ 分，被接受但比当前差得 σ₃ 分。

### 局部搜索（嵌入 ALNS 迭代中）

每次 repair 完成后，对当前解做一轮快速局部搜索：

**路径内优化**：对每个时期的每条路径，做 2-opt 改善。**路径间优化**：同一时期的两条路径之间做 relocate（将一个客户从一条路径移到另一条）和 swap（交换两条路径各一个客户）。**跨期 relocate**：将收集点 i 的一次访问从时期 t₁ 移到 t₂（需要同时检查两个时期的路径可行性和全局库存可行性）。

### 算法流程伪代码

```
S* = InitialSolution()
S = S*
T_init = ComputeInitialTemperature()
T = T_init

for iter = 1 to MAX_ITER:
    选择 destroy 算子 d (轮盘赌, 按权重 w_d)
    选择 repair 算子 r (轮盘赌, 按权重 w_r)
    
    S' = copy(S)
    d.destroy(S')
    r.repair(S')
    R4_ProductionReoptimize(S')  // 总是重新优化生产计划
    LocalSearch(S')              // 路径局部搜索
    
    ΔZ = cost(S') - cost(S)
    if ΔZ < 0 or random() < exp(-ΔZ / T):
        S = S'
        if cost(S) < cost(S*):
            S* = S
    
    更新算子得分
    if iter % segment_size == 0:
        更新自适应权重
    
    T = T * cooling_rate

return S*
```

### 实现优先级建议

如果你准备编码，我建议按这个顺序推进：

1. 先实现解的数据结构和可行性检查（含增量库存更新）
2. 实现初始解构造
3. 实现 R4（production re-optimization），这个贯穿始终
4. 实现 D1 + R1 的最简组合，跑通 ALNS 主循环
5. 逐步添加其余算子
6. 加入局部搜索
7. 参数调优（温度、冷却率、segment 大小、σ 值）
